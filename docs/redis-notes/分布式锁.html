# Redis分布式锁

👊👊

Redis分布虽然中间过程很简单，但是一些连续操作有许多的细节容易忽略

# Setnx

Setnx意为set if not exist，复合判断和创建操作

# expire复合操作

这里存在一个问题，若拿到lock的客户端后发生异常，例如宕机之类的，永远无法返回del lock了，那么就需要给lock 添加期限

setnx lock true

# 设置5秒过期时间
expire lock 5

但是setnx和expire并不是复合操作，所以应该使用

# set key value [EX seconds| PX milliseconds|KEEPTTL] [NX|XX]
set lock ture ex 5 nx

2

通过EX、PX和KEEPTTL设置过期时间单位，通过NX和XX设置执行条件

[NX|XX]

  • NX：仅当key不存在时生效
  • XX：仅当Key存在时生效

# 超时问题

Redis不能解决超时问题

若执行逻辑时间过长，锁失效已过期，其它客户端已尝试持有锁，而原持有锁客户端这是已经完成业务。出现这种情况，业务已经执行完成，代码无法严格地串行执行。

  • 建议不要用于较长时间的任务

使用随机值作为表识客户端的身份，保证持有锁客户端的锁不会被其他客户端释放。但是，匹配随机值不是原子过程，所以需要使用Lua脚本进行处理

# 可重入行

不推荐使用重入锁，因为会加重客户端复杂程度，完全可以在代码中进行调整

import java.util.HashMap;
import java.util.Map;
import java.util.Objects;

abstract class RedisReentrantLock {

   private static final ThreadLocal<Map<String, Integer>> localCounter = ThreadLocal.withInitial(HashMap::new);


   public boolean lock(String key) {
       Map<String, Integer> counter = localCounter.get();

       return counter.compute(key,
               (k, i) ->
                       i == null ?
                               tryAcquire(key) ?
                                       1 :
                                       null :
                               ++i
               ) != null;
   }

   public boolean unlock(String key) {
       Map<String, Integer> counter = localCounter.get();
       Integer count = counter.get(key);

       if (count == null) {
           return false;
       }

       if (count > 1) {
           counter.put(key, --count);
           return true;
       }


       return tryRelease(key) && Objects.equals(counter.remove(key), count);

   }


   /**
    * 尝试Redis进行加锁
    */
   protected abstract boolean tryAcquire(String key);

   /**
    * 尝试Redis进行解锁
    */
   protected abstract boolean tryRelease(String key);


   public static void main(String[] args) {

   }

}
